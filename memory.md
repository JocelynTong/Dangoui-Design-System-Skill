# 项目记忆摘要

## 当前站与 Dangoui Docs 站的关系

当前站点更像 AI / 运营 / vibecoding 使用的「风格学习与迁移工作台」：

- 主要回答：参考哪个站、该站视觉如何拆成 dangoui token、套到 demo 后是什么效果、AI 后续应该沿用哪个风格继续生成。
- 面向对象更偏运营、产品、AI 使用者、vibecoding 流程。
- 内容应尽量少讲 props、API、底层代码细节，优先展示风格选择、学习证据、token 映射和 demo 预览。

`https://dumpling.echo.tech/get-started/introduction` 这类组件库 docs 站更像「专业人员查看的组件库说明书」：

- 主要回答：组件怎么用、有哪些 props、组件行为边界、代码示例、token 规则、底层实现细节。
- 面向对象更偏开发者、设计系统维护者、需要查 API 和组件细节的 AI。
- 适合放完整组件文档、开发者说明和底层规范。

## 当前阶段的判断

目前两个站都还在雏形阶段，不急着合并，也不急着抽统一 manifest。

现在更重要的是分别验证两个站的使用场景：

- 当前站：AI 和运营到底需要看到哪些上下文，哪些内容刚好够用。
- docs 站：开发者和设计系统维护者到底需要哪些组件 API 与底层细节。
- AI：哪种 description 最能帮助它稳定选组件、套 token、生成 demo。

过早抽 manifest 可能会让文案和结构被提前锁死，导致每次微调都要考虑共享字段、展示适配和生成逻辑，反而拖慢判断。

## 维护原则

现阶段可以各自写 description，但不要重复定义事实。

可以分开写：

- 当前站写面向 AI / 运营的短 description。
- docs 站写面向开发者的完整 description。
- 两边语气、长度、重点可以不同。

尽量保持一致：

- 组件名
- token 名
- className
- props 事实
- docs URL
- 组件和 token 的真实映射关系

## 什么时候再考虑合并 manifest

出现以下情况时，再从已经跑出来的内容中抽统一 manifest：

- 同一个组件描述已经稳定，但每次都要两边同步。
- AI 因为两个描述不一致产生误解。
- docs 站和当前站都开始依赖同一批组件 metadata。
- skill 准备给更多项目或更多人正式使用。
- 维护成本已经明显变烦，而不是理论上可能会烦。

## 当前建议

短期策略：不合并 UI，不合并站点，不急着抽 manifest。

更合理的方向是：先让当前站和 docs 站各自快速试错，只把稳定事实保持一致。后续如果需要合并，再从已经验证过的内容里抽 manifest，而不是现在提前设计一个过重的数据结构。
